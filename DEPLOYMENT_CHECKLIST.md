# Production Deployment Checklist

**Purpose:** Verify system readiness before deploying Narmir Reborn to production  
**Audience:** DevOps engineers, deployment leads, engineering team  
**Last Updated:** 2026-06-29

---

## Pre-Deployment Phase (1-2 hours before)

### Code Readiness

- [ ] All commits on main branch are reviewed and merged
- [ ] Branch history clean (no stray branches, no merge conflicts)
- [ ] Latest main branch pulled: `git fetch origin main && git checkout main && git pull origin main`
- [ ] No uncommitted changes: `git status` shows clean working directory
- [ ] Lint passes: `npm run lint` returns 0 errors
- [ ] Smoke test passes locally (see CLAUDE.md smoke test section)

### Documentation Readiness

- [ ] CHANGELOG.md updated with new features/fixes
- [ ] ROADMAP.md reflects completed work
- [ ] API_ENDPOINTS.md updated if any routes changed
- [ ] Migration scripts tested locally (if database schema changed)
- [ ] All .md docs pass grammar/spell check

### Security Audit

- [ ] No secrets in code: `git log --all -S "password\|secret\|token" --oneline` returns expected results only
- [ ] SQL injection audit passed (see SUPPORT_RUNBOOK.md Item 1)
- [ ] No XSS vulnerabilities in new features
- [ ] CORS_ORIGIN correct (should be `https://narmirreborn.com`)
- [ ] Rate limiting properly configured (see RATE_LIMITING_GUIDE.md)

### Backup Verification

- [ ] Production database backed up recently: `pg_dump ... | gzip` (see BACKUP_RESTORE_RUNBOOK.md)
- [ ] Backup file tested (can restore to test DB)
- [ ] Backup stored in secure location (not Git, not public)
- [ ] Restore procedure documented and tested

## Railway Configuration Phase (30 mins)

### Environment Variables Setup

- [ ] Navigate to Railway.app → Your Project → Web service → Variables
- [ ] Verify the following variables exist (do not create new ones, only add/update shown below):

#### Critical Variables
- [ ] `JWT_SECRET` — 32+ character strong random string
  - Verify: Should NOT be visible in logs
  - Test: Generate new if unsure: `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`
  
- [ ] `DATABASE_URL` — Auto-generated by Railway from PostgreSQL service
  - Verify: Should be `postgresql://...@....railway.internal:5432/...`
  - Verify: PostgreSQL service is linked (Web service → Dependencies)
  - Test: Can connect via `railway psql`

#### Required for Production
- [ ] `CORS_ORIGIN` — Exact match of frontend domain
  - Value: `https://narmirreborn.com`
  - Verify: No trailing slash, HTTPS protocol
  
- [ ] `NODE_ENV` — Auto-canonicalized to `production` by index.js line 8
  - Value: Railway shows as `Production` (environment name)
  - Note: We normalize this in code, no manual action needed

#### Strongly Recommended
- [ ] `ADMIN_SECRET` — 16+ character strong random string
  - Generate: `node -e "console.log(require('crypto').randomBytes(16).toString('hex'))"`
  
- [ ] `CONFIRM_SECRET` — 16+ character strong random string
  - Generate: `node -e "console.log(require('crypto').randomBytes(16).toString('hex'))"`

#### Optional (Only if Using)
- [ ] `DISCORD_BOT_TOKEN` — If Discord integration enabled
- [ ] `DISCORD_UPDATES_CHANNEL_ID` — If Discord integration enabled
- [ ] `DISCORD_BUG_REPORTS_CHANNEL_ID` — If Discord integration enabled
- [ ] `ADMIN_ALLOWED_IPS` — If restricting admin endpoints to specific IPs

### PostgreSQL Verification

- [ ] PostgreSQL service is running (green status in Railway dashboard)
- [ ] PostgreSQL is linked to Web service (Web → Dependencies)
- [ ] Can connect via Railway CLI: `railway psql` (opens psql prompt)
- [ ] Database name matches expected (usually `railway`)
- [ ] No pending migrations to run

### Domain & SSL Setup

- [ ] Domain `narmirreborn.com` points to Railway (via DNS/Route)
  - Verify: In Railway, go to Settings → Domains
  - Add domain: `narmirreborn.com`
  - Railway auto-provisions SSL certificate (Let's Encrypt)
- [ ] SSL certificate generated: Railway shows green checkmark
- [ ] Test HTTPS works: `curl -I https://narmirreborn.com` returns 200
- [ ] HTTP redirects to HTTPS (configured in index.js middleware)

## Pre-Production Test Phase (1 hour)

### Staging Environment (Optional but Recommended)

If you have a staging Railway environment:

- [ ] Deploy to staging first
- [ ] Run smoke tests against staging
- [ ] Test critical user flows (auth, game start, turn submission)
- [ ] Monitor staging logs for errors
- [ ] Performance acceptable (response times <1s for most endpoints)

### Local Simulation Test (Minimum)

```bash
# Simulate production environment locally
export NODE_ENV=production
export JWT_SECRET=$(node -e "console.log(require('crypto').randomBytes(32).toString('hex'))")
export DATABASE_URL="postgresql://postgres:smoke@localhost/narmir_smoke"
export CORS_ORIGIN="http://localhost:3000"  # local test origin
export ADMIN_SECRET=$(node -e "console.log(require('crypto').randomBytes(16).toString('hex'))")
export CONFIRM_SECRET=$(node -e "console.log(require('crypto').randomBytes(16).toString('hex'))")

# Start server
node index.js &
sleep 4

# Test baseline endpoints
curl -s http://localhost:3000/api/forum/boards | jq .
curl -s http://localhost:3000/api/auth/me | jq .
curl -s http://localhost:3000/portal | grep -q "portal-root" && echo "✅ Portal loads"
curl -s http://localhost:3000/game | grep -q "main.jsx" && echo "✅ Game client loads"

# Test rate limiting (should block after 10 requests)
for i in {1..15}; do
  curl -s http://localhost:3000/api/auth/login -X POST | jq '.error' 2>/dev/null
done

# Kill server
kill $(lsof -t -i:3000)
```

Expected results:
- [ ] Forum boards API returns valid JSON
- [ ] Auth/me returns "Not authenticated" (no token)
- [ ] Portal renders successfully
- [ ] Game client loads
- [ ] Rate limiting engages on excessive requests

## Deployment Phase (15 mins)

### Option A: Manual Deploy via Railway Dashboard

1. [ ] Navigate to Railway.app → Your Project → Web service
2. [ ] Click "Deploy" button or "View logs"
3. [ ] Watch deployment progress
4. [ ] Deployment completes (should see ✅ icons)
5. [ ] Logs show:
   - `[secrets] ✅ All required secrets configured`
   - `[db] ✅ PostgreSQL connected successfully`
   - `[server] ✅ Listening on port 3000`
6. [ ] No error logs (filter for `[ERROR]` or `❌`)

### Option B: Deploy via GitHub Push

1. [ ] Merge feature branch to main: `git merge <branch> main`
2. [ ] Push to origin: `git push origin main`
3. [ ] Railway auto-detects push, starts deployment
4. [ ] Monitor via Railway dashboard (same logs as Option A)

### Option C: Deploy via Railway CLI

```bash
railway login
railway link  # Select your project
railway deploy  # Deploy current branch
```

Watch logs during deployment.

## Post-Deployment Validation (30 mins)

### Immediate Checks (within 5 minutes)

- [ ] Application is running (green status in Railway)
- [ ] No crash loops (logs show clean startup)
- [ ] Secrets validated on startup:
  - Check logs for: `[secrets] ✅ All required secrets configured`
  - Check logs for: `[db] ✅ PostgreSQL connected successfully`
- [ ] Server is listening:
  - Check logs for: `[server] ✅ Listening on port 3000`

### HTTP Connectivity (within 10 minutes)

- [ ] Frontend loads: `curl -I https://narmirreborn.com/` returns 200
- [ ] API responds: `curl https://narmirreborn.com/api/auth/me` returns JSON
- [ ] CORS headers present: 
  ```bash
  curl -I -H "Origin: https://narmirreborn.com" https://narmirreborn.com/api/auth/me
  # Should see: Access-Control-Allow-Origin: https://narmirreborn.com
  ```
- [ ] HTTPS enforced: `curl -I http://narmirreborn.com/` redirects to HTTPS (301/302)

### Core Functionality (within 15 minutes)

Run these tests against production:

```bash
# Test 1: Forum works
curl -s https://narmirreborn.com/api/forum/boards | jq '.data | length' | grep -q '[0-9]'
echo "✅ Forum test passed"

# Test 2: Auth works
curl -s https://narmirreborn.com/api/auth/me | jq '.authenticated' | grep -q 'false'
echo "✅ Auth test passed"

# Test 3: Market data loads
curl -s https://narmirreborn.com/api/market/prices | jq '.data | length' | grep -q '[0-9]'
echo "✅ Market test passed"

# Test 4: Rankings work
curl -s https://narmirreborn.com/api/rankings | jq '.data | length' | grep -q '[0-9]'
echo "✅ Rankings test passed"
```

### Performance Check (within 20 minutes)

- [ ] Response times normal:
  ```bash
  curl -w "@curl-format.txt" -o /dev/null -s https://narmirreborn.com/api/forum/boards
  # Should see total_time < 1000ms (1 second)
  ```
- [ ] No 503 errors in logs
- [ ] Database connection pool healthy (check logs or monitoring)
- [ ] No memory leaks (check logs for garbage collection)

### Log Inspection (within 25 minutes)

- [ ] Railway logs show no ERROR or CRITICAL messages
- [ ] No secret exposure in logs (SecretsManager sanitizes)
- [ ] No SQL query errors
- [ ] Rate limiting logs show normal activity (no excessive blocks)

## Rollback Procedure (If Needed)

If deployment has critical issues:

### Option 1: Deploy Previous Main Branch

```bash
git checkout main
git reset --hard origin/main~1  # Go back one commit
git push -f origin main  # Force push to trigger redeploy
```

⚠️ Only use if new code has critical bug. Avoid if data consistency might be affected.

### Option 2: Database Rollback

If database migration was the issue:

1. [ ] Stop production server (pause Railway service)
2. [ ] Restore database from backup:
   ```bash
   # Using backup script
   ./scripts/restore-database.sh <backup-file>
   ```
3. [ ] Restart server
4. [ ] Verify data consistency

### Option 3: Downtime Message

If neither option works:

1. [ ] Update status page (if you have one)
2. [ ] Post to Discord: "We're experiencing issues and working on it"
3. [ ] Begin incident investigation
4. [ ] Roll back to last known good deployment

## Incident Response

### If Deployment Fails at Configuration Phase

**Error:** Variables tab missing DATABASE_URL

**Action:**
1. Go to PostgreSQL service → Links
2. Click "Link to service" → select Web service
3. Railway regenerates DATABASE_URL
4. Retry deployment

---

### If Deployment Fails at Startup

**Error:** `[secrets] ❌ Server startup blocked due to missing secrets`

**Action:**
1. Check Variables — which secret is missing?
2. Add the secret with strong value
3. Redeploy

**Error:** `[db] ❌ PostgreSQL connection failed`

**Action:**
1. Verify PostgreSQL service is running (green status)
2. Check DATABASE_URL format: should be `postgresql://...@....railway.internal:5432/...`
3. Verify connection pool settings in index.js
4. Restart PostgreSQL service and redeploy

---

### If API Responds But Fails Tests

**Error:** CORS errors on frontend

**Action:**
1. Check CORS_ORIGIN matches exactly (including protocol, no trailing slash)
2. Verify frontend domain matches CORS_ORIGIN
3. Browser may have cached old response — test incognito window
4. Clear Cloudflare cache (if used)

**Error:** Rate limiting too aggressive

**Action:**
1. Check rate limiter config (config/rate-limiting.js)
2. Temporarily reduce thresholds during testing
3. Monitor rate limited requests in logs
4. Adjust and redeploy
