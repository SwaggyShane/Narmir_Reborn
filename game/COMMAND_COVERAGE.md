# Mutator coverage — CommandHandler vs. domain modules

**Date:** 2026-07-19 (route tables), policy text refreshed 2026-07-22 (`docs/dev/MUTATOR_POLICY_PLAN.md`
M1). Generated by scanning every mutating (`POST`/`PUT`/`PATCH`/`DELETE`) route across
`routes/kingdom-*.js` and `routes/hero.js` for `commandHandler.handle({ type: … })` calls,
cross-referenced against `game/command-handler.js`'s actual `COMMAND_TYPES` and `handle()`
switch. **The route tables below are a snapshot and have a known small drift** (M0 recon
recounted 82 mutating routes across `kingdom-*`+`hero` against this doc's stated 83 — exact
re-list is M3, not done here). Re-run the classification if routes change; do not hand-edit
the tables without re-verifying.

**Where simulation code lives ≠ which façade routes use.** `CommandHandler` is a route-facing
façade for classic sim verbs — it does not require that logic sit in `game/engine.js`. After
the engine extract (`docs/dev/ENGINE_EXTRACT_PLAN.md`, complete 2026-07-22), the turn pipeline
lives in `game/lib/turn-pipeline.js` (+ `game/lib/turn-*.js` phase modules), expedition/epic-trek
resolution in `game/lib/expedition-resolution.js`, and region resolution in
`game/lib/region-resolution.js`. `game/engine.js` itself is now a 526-line composition root +
re-export barrel, not where the logic runs.

## Policy: A / B / S

Three shapes, not one rule. Classify by what the mutator *is*, not by habit.

### Policy A — CommandHandler path

**When:** Classic sim actions that fit `handle(type, payload) → { updates, events }` (or a
small fixed result shape) and are already (or naturally) wired through `COMMAND_TYPES`.

**Live today:** turn, combat, spell, covert-\*, expeditions, hire/recruit, build-queue actions
(queue/demolish), study/select-school, purchase-upgrade, calculate-score, raid-trade-route,
forge-tools (legacy smithy hammers/scaffolding — **not** the Forge & Lava Industry system),
award-xp/troop-xp, prestige (fenced — see below).

**Rules:**

- Routes must not call forbidden `engine.*` mutators (`npm run check:command-boundary`).
- Prefer `commandHandler.handle(...)`.
- Simulation may live in the `engine` barrel, `turn-pipeline`, `game/lib/*`, or a domain
  module — CommandHandler is the route-facing façade, not a requirement that the function
  still physically sit in `engine.js`.

### Policy B — Domain module + route transaction

**When:** Multi-step atomic flows, bespoke persistence, or already-modular systems that don't
fit CommandHandler's simple shape.

**Live today:** Forge & Lava Industry (`game/forge-upgrades.js`, `game/forge-production.js`,
`game/flux-barge.js`, `game/lava-vents.js`, `game/lava-expedition.js`), Dragon Evolution
(`game/evolution/`), Prestige rebirth (`game/prestige/`), attunements/synergies
(fragment-bonus-manager + related managers), most allocations, market/bank, many
build/exploration helpers.

**Rules:**

1. The route never reaches into `game/engine.js`'s forbidden mutators (same
   `check:command-boundary` gate as Policy A).
2. **Single domain home** — a real, single-purpose module, not scattered inline route logic.
3. Multi-write mutations wrap in `db.withTransaction` (or the route documents why a single
   statement is already atomic).
4. **Response contract (companion rule, separate track):** where the client expects store
   updates, the route should return domain-shaped `updates` via `structureUpdates()` /
   `apiCallAndSync()`. Tracked separately — see `docs/dev/MUTATOR_POLICY_PLAN.md` §9; not
   re-litigated per-route in this doc.
5. Document the system in the coverage matrix below.

**Why not force everything through `CommandHandler` instead (the alternative, rejected):**
Prestige rebirth is the concrete precedent for why not — `game/command-handler.js`'s
`handlePrestige()` deliberately throws (`"Prestige rebirth must use POST /api/kingdom/rebirth
(atomic wipe). CommandHandler prestige is disabled."`) rather than dispatching, because
rebirth's `FOR UPDATE` row lock + atomic multi-table wipe transaction doesn't fit
`CommandHandler`'s simple `handle(type, payload) → { updates, events }` shape without
`CommandHandler` itself growing transaction-management responsibility it doesn't otherwise
have. Retrofitting Forge/Evolution/Prestige/attunements into `COMMAND_TYPES` would be a
large, purely mechanical migration of already-correct, already-tested code for no behavior
change and real regression risk — rejected per "no path of least resistance" *in the other
direction*: forcing a one-size-fits-all mechanism onto systems that already have a clean,
correct, differently-shaped boundary is not actually simpler, just uniform for its own sake.

### Policy S — System / tick (not player HTTP mutators)

**When:** Server jobs with no player "command verb" shape, or not player-initiated HTTP —
the regen timer, `resolveRegions` on tick, boot repair, scheduled audits.

**Rules:** Must not reintroduce player combat/spell/covert as raw socket mutators bypassing
HTTP + CommandHandler (that failure mode is exactly what the socket exception below used to
be, before A5-4/A5-5 closed it).

### Other (label, don't force into A/B/S)

`alliance.js` (7 mutating routes), `forum.js` (13 mutating routes), and admin routes (47
mutating routes across `routes/admin*.js`) are **not currently classified** in the matrix
below — they weren't in scope for the 2026-07-19 kingdom-\*/hero pass. Flagged by M0 recon;
classifying them is M3's job, not retrofitted here.

### Decision tree (new mutating endpoint)

```text
Is this a player HTTP mutator?
├─ No → Policy S (regen, boot, admin-internal jobs) or Other / N/A
└─ Yes
   ├─ Needs multi-table atomic TX or domain already owns TX shape?
   │  └─ Yes → Policy B (domain module + route txn); list in coverage matrix
   ├─ Fits CH handle(type) → updates/events and is classic sim?
   │  └─ Yes → Policy A (COMMAND_TYPES + handle)
   └─ Unsure → gray-area log entry (docs/dev/MUTATOR_POLICY_PLAN.md M6); default to B if a
      domain module exists, else discuss
```

**Default for brand-new greenfield features:** prefer Policy B if the feature owns its own
tables/modules (Forge-like); prefer Policy A if it's another verb on the shared kingdom sim
result bag (attack / spell / turn-like).

**Socket exception — closed, not open (corrected 2026-07-22, was stale since A5-4/A5-5 landed
2026-07-19 without this doc being updated).** This section previously claimed `game/sockets.js`
calls `engine.resolveMilitaryAttack`/`engine.castSpell` directly and that `check:command-boundary`
doesn't scan it. **Neither is true anymore.** A5-5 (2026-07-19) deleted the socket
attack/spell/covert handlers outright (~340 lines — the shipped client never emitted those
socket events in the first place, so nothing depended on them) rather than migrating them, and
A5-4 (same day) added `game/sockets.js` to `check:command-boundary`'s `STRICT_FILES_ABSOLUTE`
scan list. Verified 2026-07-22: `grep` for `engine.resolveMilitaryAttack`/`engine.castSpell` in
`game/sockets.js` returns zero matches, and `npm run check:command-boundary` reports "scanned 24
files (23 routes/, **1 other**)" — that one other file is `game/sockets.js`, clean. Real combat/
spell/covert continue to work correctly via HTTP + CommandHandler, untouched by A5-5, still
covered by `test:systems`'s `combat`/`spells`/`covert` suites.

**Dead command types removed as part of this pass:** `process-build-queue` — confirmed zero
real callers, redundant (build-queue advancement already happens automatically every turn
inside `processTurn`, `game/lib/building-research.js`'s `processBuildQueue(k, events, xpSourcesAccum)`
call — this citation originally said `game/construction.js`, an orphaned pre-refactor duplicate
deleted 2026-07-19; corrected here), and **actively broken** if it had ever been called: `handleProcessBuildQueue(kingdom,
db)` called `engine.processBuildQueue(kingdom, db)`, passing the DB adapter where the real
function signature expects an `events` array — would have thrown a `TypeError` on first use.
Removed from `COMMAND_TYPES`, its `case`, and its handler method.

**Kept as intentional, not dead:** `prestige` stays in `COMMAND_TYPES` — it is a working
fence, not unused code the way `process-build-queue` was. See the policy section above.

## Coverage matrix

`CH:<type>` = routed through `commandHandler.handle({ type: '<type>' }, …)`.
`direct+txn` = calls its own domain module(s) directly, wrapped in `db.withTransaction`.
`direct` = calls its own domain module(s)/`db` directly, no explicit multi-statement
transaction wrapper found — for most of these that's correct (a single `UPDATE` via
`applyUpdates`/`db.run` is already atomic at the DB level on its own); this column is
descriptive, not a verdict that anything here is unsafe. A real transaction-safety audit
of the `direct` rows is separate, follow-on work, not part of this pass.

## Summary

- 83 mutating routes across `kingdom-*` + `hero` (2026-07-19 count; M0 recon 2026-07-22
  recounted 82 — exact re-list pending M3, do not treat either number as final). 13 route
  through `CommandHandler`. 70 mutate via their own domain module + route-level transaction —
  this is the expected, policy-sanctioned shape for everything except the classic Policy A
  systems above.
- Zero routes reach into `game/engine.js`'s forbidden mutators directly (`check:command-boundary`
  already enforces and confirms this) — including `game/sockets.js`, which is outside this
  table entirely (it's not a kingdom route) but has been scanned since A5-4 and has carried
  zero forbidden calls since A5-5 deleted the code that made them. **No known gap remains
  here** as of 2026-07-22.
- `alliance.js`, `forum.js`, and admin routes are real mutating surfaces not yet covered by
  this table — see "Other" above. Not a gap in what's covered so much as scope this pass
  never claimed.
